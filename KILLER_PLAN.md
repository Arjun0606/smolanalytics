# THE MIXPANEL-KILLER MASTER PLAN (2026-07-15)

## definition of killer

Mixpanel-killer, for the 1-20-person product-team wedge, is a falsifiable state, not a feature count. Three pillars, each independently testable: (1) CAPABILITY — the team's actually-used core reproduces: Mixpanel's own migration doc concedes '<20% of the data is used for 80%+ of queries' (docs.mixpanel.com/docs/migration), so the floor is exactly BAR-1: trends (totals + range-deduped uniques + property measures, hour/day/week/month, multi-series), funnels (ordering modes, exclusions, per-step filters, window, breakdown, exact time-to-convert), retention (D1/D7/D30 day+week triangles, on and on-or-after modes), windowed breakdown, saved cohorts usable as a filter in every report, boards with a board-level date range, the datapoint→users→cohort drill loop, per-report CSV, Slack alerts+digests, identity aliasing. Nothing beyond it: messaging, experiments, surveys, autocapture-by-default, JQL-style query languages are features Mixpanel itself unshipped as bloat. (2) CORRECTNESS — every number is governed by a published per-report counting contract (CONTRACTS.md), each contract is a named CI test, the engine is deterministic (same dataset + params + frozen clock → byte-identical JSON across runs and replicas), never samples, and is self-consistent: one question yields ONE number across dashboard, /v1 HTTP, MCP, and ask, across every time grain. Because we control all six discrepancy causes in Amplitude's own checklist (amplitude.com/docs/faq/data-discrepancies-with-other-platforms), our dual-write acceptance bar is exact equality on totals (0 delta) and exact-or-ledger-explained on uniques — we deliberately forfeit the industry's 5% folklore tolerance; a no-sampling engine that claims determinism cannot hide behind it. (3) TRUST — the switch runs on Mixpanel's own playbook terms: one engineer, one sprint (1-3 weeks), importers over both incumbents' free raw-export APIs with tagged-slice validation, a parallel-run reconciliation report that classifies every delta against a public divergence ledger, provenance receipts that cite only CI tests that actually exist, and zero churn-trigger replication (no free-tier cliff, no sales wall on export/alerts/pipelines, no plan-gated sync cadence, no silent data mutation). KILL CONDITION: a Mixpanel power user runs the 30-minute evaluation checklist against a live instance and scores 13/13, then dual-writes via Segment for two weeks and their #1 funnel reproduces its known number at full unthrottled volume with every residual delta explained by the ledger. Today we score 5/13 and fail the self-consistency audit against ourselves (three answers — 292/290/290 — for 'signups last 7 days', and a breakdown that ignores its window) before a Mixpanel comparison would even begin. The phases below close the gap in the order evaluators actually check.

Semantics: internally exact by construction; diverges from Mixpanel calendar-day presets by design (see divergence ledger D6).

- WINDOW-2 (bucket-sum invariance): for every event, window, and filter set, Σ bucket counts is identical across interval ∈ {hour, day, week, month} and equals the unbucketed total under totals math. Violated today: 292 (day) vs 290 (week), same second, live demo. CI: randomized property test over generated datasets + a live-demo curl pair in the release checklist.

- SURFACE-1 (agreement everywhere): ask(Q).number == GET /v1/<report>(params(Q)) == MCP tool == dashboard JSON for every window phrase ask supports, at a frozen clock. - TRENDS-TOTAL: total = count of qualifying event instances in the resolved window, no per-user dedup. Matches Mixpanel Insights 'Total count of [event] performed' (docs.mixpanel.com/docs/reports/insights). Passes today; pin in CI.

- TRENDS-UNIQUE: per-bucket points = distinct users active in that bucket (a user may appear in multiple buckets — matches Mixpanel line-chart per-interval behavior; note the doc quote for this was misattributed per cross-exam, so this leg is anchored to our fixture, not their docs). The range total under unique=true = distinct users deduplicated across the WHOLE window (Mixpanel/Amplitude range-uniques semantics) — NOT Σ per-bucket uniques, which today reports 397 = Σ[52,49,54,53,47,54,57,31] and silently inflates. Under STABILITY.md additive rules the corrected number ships and the old sum, if kept, gets its own named field (bucket_unique_sum). CI fixture: one user active on 2 days → points [1,1], total 1.

- TRENDS-MEASURE: measure ∈ {sum, avg, min, max, median, p90} over the numeric property across qualifying event instances per bucket. median/p90 are EXACT order statistics with a pinned rule (median = mean of the two central order statistics for even n; p90 = value at ceil(0.9·n)-th order statistic), never approximate. Empty bucket → null, not 0 (0 is a legitimate measurement; PostHog coerces ifNull(...,0) and its own code comment admits 'not quite kosher' — divergence D13). Matches Mixpanel Aggregate Property vocabulary minus their approximate percentiles (divergence D2).

- TRENDS-MULTISERIES: events=a,b,... (≤8) computes each series independently under identical window+filters; every series independently satisfies WINDOW-1/2 and TRENDS-UNIQUE. Live today (query_api.go:226-237); pin with a fixture asserting per-series totals equal single-event queries.

- TRENDS-XAU (Phase 4): DAU/WAU/MAU as a trends measurement = distinct users in the rolling half-open window (point − 24h|7d|30d, point] per plotted point — matches Mixpanel XAU (docs.mixpanel.com/docs/reports/insights) and PostHog's implementation contract ('the 7 days leading up to the label, non-inclusive'). No silent semantics rewrite at coarse grains (PostHog silently converts WAU→DAU when interval ≥ week — we 400 on incompatible grain instead, divergence D12 family).

- BREAKDOWN-WINDOW: /v1/breakdown honors days/from/to identically to trends. Violated today — apiBreakdown returns all-time counts for every value of days (1822 for days=1/7/30 while trends 7d total is 532). This is the first reconciliation check any evaluator runs; fix is the Phase 0 headline.

- BREAKDOWN-ADD (segment additivity): Σ segment counts including '(none)' == the unsegmented total for identical window+filters, on trends, funnel steps, and /v1/breakdown; events missing the property land in '(none)', never dropped. Funnel already passes (412+244=656; 660+380=1040, verified live) — pin it; breakdown becomes verifiable once BREAKDOWN-WINDOW lands. CI: randomized property test.

- BREAKDOWN-RECIP (filter↔breakdown reciprocity): trends total with f=prop:eq:V == segment V's count in breakdown; f=prop:regex:R total == Σ matched segments. Passes today (380 mobile; 469 = 233 google + 236 reddit for ^(google|reddit)$, verified live). Pin in CI — Mixpanel evaluators run this constantly.

- BREAKDOWN-OTHER: when distinct segment values exceed the limit (default 25), overflow rolls into 'Other' = exact sum of the remainder, ranked by count desc with ties broken by value asc (deterministic); 'Other' is a true sum so BREAKDOWN-ADD still holds exactly. Comparable to PostHog BREAKDOWN_VALUES_LIMIT=25 semantics but with additivity guaranteed.

- FUNNEL-ARITH: step counts non-increasing; conversion_from_top_i = c_i/c_0; conversion_from_prev_i = c_i/c_(i−1); dropped_i = c_(i−1)−c_i; overall = c_last/c_0; identical arithmetic inside every breakdown segment. Verified exact on the live demo (1040→656→294; 0.63077; 0.44817; 0.28269). Pin in CI.

- FUNNEL-ENTRY (default counting = uniques, best-anchor): each user counts at most once, at their furthest-reached run; the anchor is the Step-1 occurrence yielding the furthest progression, ties broken by earliest anchor — this is what internal/funnel/funnel.go furthestStep (lines 152-179) already implements and it matches Amplitude's documented 'earliest and longest conversion' rule (amplitude FUNNEL-3) and the spirit of Mixpanel Optimized Re-entry. entry=first ships as a compat param implementing Mixpanel's DEFAULT ('conversion window starts on the first instance of the Step 1 event... will not be updated by later instances' — funnels-advanced) for parallel-run parity. CI fixture: Step1(d0), Step1(d20), Step2(d20), window 7d → converted under default, NOT converted under entry=first. Default choice justified in divergence D10.

- FUNNEL-WINDOW: every matched step must satisfy event.ts − anchor.ts ≤ window (whole-funnel window from entry — all three incumbents agree); boundary INCLUSIVE: an event at exactly anchor+window converts. Mixpanel leaves inclusivity undocumented (confirmed ambiguity); PostHog implements <= (funnel base.py generates 'latest_i <= latest_0 + INTERVAL window'); we pin inclusive and document it. CI fixtures at exactly t+window and t+window+1s.

- FUNNEL-RANGE: the date range gates funnel ENTRY (a qualifying Step-1 in range); completion may occur after the range ends, within the window — the engine scans events past `to` by up to window for funnel queries. Matches Mixpanel's directional community semantics (entry-gated, completion-after-range); today our engine range-scopes ALL events so late completions are lost. CI fixture: Step1 on the last in-range day, Step2 two days later, window 7d → converted.

- FUNNEL-ORDER: order=ordered (default; loose subsequence — foreign events allowed between steps; matches Mixpanel loose order and Amplitude this-order), order=strict (step i+1 must be the literally-next event after step i; ANY intervening event kills the run — matches PostHog strict), order=unordered (all steps within window of anchor, any sequence — matches Amplitude any-order / PostHog unordered). Already implemented in internal/funnel/funnel.go:184-205 with ParseOrder 400-ing unknowns; exposure is Phase 1. Same-timestamp rule: two DIFFERENT steps sharing one timestamp are interchangeable (either order satisfies; matches Amplitude same-second and PostHog same-timestamp handling at the 0s limit); one physical event never satisfies two steps. NO multi-second grace period (divergence D3).

- FUNNEL-EXCLUDE: exclusions=e1,... — a user who fires an exclusion event between their counted run's anchor and its final matched step is removed from the ENTIRE funnel (all step counts). This is PostHog-compatible ('These people are completely excluded from the entire funnel') and is what internal/funnel/funnel.go:207-213 already implements; it deliberately diverges from Mixpanel's attempt-killing-with-re-entry (divergence D8). An exclusion event equal to any step's event returns 400 (PostHog validation rule matched). CI fixtures: exclusion inside the run → user absent from every step; exclusion before Step-1 or after conversion → no effect.

- FUNNEL-STEPFILTER: per-step property filters restrict which events match a step (stepMatches, funnel.go:218-232). Pinned interplay: an event matching step i's NAME but failing its filter does not match step i, and in strict mode it still counts as an intervening event (kills the run) because no step consumed it — deterministic and PostHog-consistent. Mixpanel's 'inline filters are performed pre-query' wording is honored for ordered/unordered modes where non-matching events are invisible anyway. CI fixtures for both ordered and strict.

- FUNNEL-TTC: time-to-convert (median today, p90 added) computed over counted converting runs' anchor→final-step durations, EXACT order statistics, deterministic. Divergence D2 from all three incumbents' approximate percentile algorithms (Mixpanel's docs verbatim: 'can be non-deterministic'). CI: hand-computed fixture.

- FUNNEL-BREAKDOWN-ATTR: a user's segment = the property value on their EARLIEST in-range Step-1 event, missing → '(none)' (current implementation, funnel.go:100-130), even when the counted run anchors at a later Step-1 — one stable segment per user, so BREAKDOWN-ADD holds exactly (Σ per-segment step counts == unsegmented counts). Analogous to Mixpanel first-step attribution / PostHog first_touch default; per-step and last-touch attribution modes are explicit non-goals until demanded. CI fixture where a user's Step-1s carry different property values.

- RETENTION-COHORT: cohort bucket = calendar interval (day: midnight-aligned; week: ISO Monday; month: 1st — in the project timezone once TZ-1 lands, UTC until then) containing a qualifying start event; recurring cohorting — a user appears in EVERY bucket where they fired the start event, counted once per bucket (Mixpanel verbatim: 'counted once per bucket, but can be included in more than one bucket'); cohort size = distinct users. Week anchor MUST equal the trends week anchor (violated today: trends weeks are Monday calendar weeks, retention weeks are range-relative Thursday-anchored chunks — fails Amplitude checklist Q4 against ourselves).

- RETENTION-RETURN: default mode=on — retained in cell k iff a return event falls in bucket (birth_bucket + k) exactly; cells are independent and historically stable (Amplitude/PostHog norm). mode=on_or_after — retained in cell k iff a return event falls in ANY bucket ≥ k, rows monotone non-increasing (Mixpanel's DEFAULT; exposed for parallel-run parity, not our default — divergence D7). Cell values = distinct users. k=0 column = cohort size shown as 100% by definition (the anchor, matching today's demo); return criteria evaluate from k=1 — Mixpanel's bucket-0 behavior is undocumented (confirmed ambiguity), ours is definitional and stated. CI: fixtures for both modes over one dataset, asserting on_or_after row-monotonicity and on-mode cell independence.

- RETENTION-ROLLING: rolling=true anchors each user at their first qualifying start event t0; cell k = distinct users with a return event in [t0 + k·U, t0 + (k+1)·U) where U = 24h/7d/30d for day/week/month buckets (PostHog 24_hour_windows-compatible; Mixpanel 'rolling' Day 1 = 24-48h matched). CI test vector = Mixpanel's own documented example: start Mon 8pm, return Tue 6am → NOT Day-1 retained under rolling, IS Day-1 retained under calendar mode.

- RETENTION-SUMMARY: the headline retention curve = cohort-size-weighted average over COMPLETE buckets only (weight_i = cohort_size_i / Σ sizes; rate = Σ(retained_i·w_i)/Σ(total_i·w_i)); in-flux buckets carry incomplete:true and are excluded from the average — Mixpanel's weighted-average + completed-periods rule matched verbatim. retention.Summarize stays the single shared implementation across HTTP and MCP (query_api.go:327-331) so surfaces cannot drift.

- COHORT-DEF: cohort criteria = boolean AND/OR tree over {performed E [count op N] in window W; did NOT perform E in W; property op value using the existing 11-op ?f grammar}; windows = rolling last-N-days or absolute range (a deliberate subset of Amplitude's define-cohort vocabulary). Membership evaluates dynamically at query time over the current event log, deterministically; cohorts are saveable from any report segment and usable as a ?f filter in EVERY report; GET /v1/cohorts/{id}/users returns exact membership (route exists, api.go:222). CI round-trip: report segment → save cohort → filter any report by cohort reproduces the originating count exactly.

- IDENTITY-1: alias(anon_id → user_id) retroactively re-attributes ALL of anon_id's events to user_id for every subsequent query, immediately (no ≤24h propagation — divergence D4); canonical id = user_id (Mixpanel Simplified: 'canonical distinct_id is always the $user_id'); unlimited anon ids per user; exactly ONE user_id per cluster; user↔user merges rejected with 400 (Mixpanel Simplified cluster rules + Amplitude's 'can't merge user IDs' matched). Uniques everywhere count post-merge canonical identities, so a merge retroactively shrinks historical unique counts — PostHog's documented Alice-example behavior, adopted and stated. CI: uniques-before/after-alias fixture.

- INGEST-DEDUP: for events carrying insert_id, duplicates iff (name, timestamp, distinct_id, insert_id) are all identical (Mixpanel's exact dedup tuple matched — reference/import-events); exactly one survives in every query; unbounded lookback; deterministic winner = last-ingested (Mixpanel query-time rule 'most recent version shown' matched; their nondeterministic same-calendar-day compaction pass is NOT replicated — divergence D11). No insert_id → no dedup. Millisecond timestamp precision preserved ingest→query→export (Mixpanel exports at second precision; we never degrade).

- NO-SAMPLING (covenant): no query path computes over a subset of matching events; no sampling parameter exists anywhere in the API or MCP surface; CI determinism harness asserts identical dataset + params + frozen clock → byte-identical JSON across runs and across replicas. Anti-matched against Mixpanel Enterprise query-time sampling (10% of users, upsampled) — divergence D1.

- SESSIONS-1 (Phase 4): sessions are virtual, computed at query time — a session = a maximal run of one user's events where consecutive gaps ≤ timeout (default 30 min, configurable 1min-24h), hard 24h duration cap, NO midnight split (divergence D9); changing the timeout retroactively recomputes all history (Mixpanel's query-time-virtual property matched — features/sessions); 'sessions containing event E' becomes a trends measurement (Mixpanel 'Sessions with Event' matched). CI: fixture with a 29-min and a 31-min gap; retroactive-redefinition test.

- TZ-1 (Phase 4): storage is always UTC; a single project timezone converts ONLY at query output (bucket boundaries, window resolution, labels); changing it re-renders all history with zero stored-data mutation (Mixpanel ≥2023 project model matched — orgs-and-projects/managing-projects); default UTC; every response echoes the timezone it resolved against. CI: same dataset queried under two timezones shifts bucket boundaries but preserves window-wide totals.

## deliberate divergences from mixpanel (the ledger)
- D1 — No query-time sampling, ever (Mixpanel Enterprise samples a uniform 10% of users and upsamples user-scaling aggregations; docs.mixpanel.com/docs/reports). WHY: determinism is the product; a sampled answer cannot carry an exactness receipt, and our wedge segment never has the >2B-events/mo problem sampling exists to solve.
- D2 — Exact medians and percentiles via full order statistics (Mixpanel: 'approximate algorithm... can be non-deterministic', funnels-advanced; PostHog: ClickHouse approximate median). WHY: provenance receipts require reproducible numbers; at wedge-segment data volumes exact stats are computationally trivial, so approximation is pure downside.
- D3 — No 2-second funnel grace period (Mixpanel makes consecutive steps ≤2s apart interchangeable; Amplitude rounds to 1s and accepts either order). We treat only IDENTICAL timestamps as interchangeable. WHY: multi-second clock-skew guessing is nondeterministic magic; the parallel-run delta is bounded and enumerable (reversed pairs 1-2000ms apart) and the reconciliation report classifies it explicitly.
- D4 — Identity merges apply immediately and atomically at query time (Mixpanel: 'may take up to 24 hours' to reflect in reports). WHY: eventual consistency in an analytics store is silent-wrongness during exactly the window when a developer is testing their integration; a single-binary engine can afford read-time correctness.
- D5 — unique=true range total = range-deduplicated distinct users, with the per-bucket series separately named (our current 'total' silently sums per-bucket uniques: 397 = Σ buckets, inflated vs true range uniques). WHY: one field name must have one meaning; Mixpanel/Amplitude evaluators read 'total uniques over 7 days' as range-deduped and will flag the inflation in any parallel run.
- D6 — days=N means the strict rolling half-open window (now − N×24h, now], echoed as resolved from/to in every response (Mixpanel presets = whole calendar days incl. partial today, in project timezone). WHY: either convention is defensible — the crime we're fixing is having BOTH meanings under one label in one product; we pick one, echo the resolved bounds, and offer calendar-aligned from/to for preset parity.
- D7 — Retention default = mode=on with independent, historically stable cells (Mixpanel's DEFAULT is on-or-after, where a future return retro-fills earlier cells, so yesterday's chart mutates over time). WHY: numbers that silently change after being read are user-hostile for a tool whose covenant is reproducibility; on_or_after ships as a one-param Mixpanel-parity mode for parallel runs.
- D8 — Funnel exclusions remove the user from the entire funnel (PostHog-compatible 'completely excluded'; already implemented at internal/funnel/funnel.go:207-213) rather than Mixpanel's kill-current-attempt-allow-re-entry. WHY: our uniques model counts one best run per user, so attempt-scoped exclusion is ill-defined without totals/re-entry counting; full-drop is deterministic, simpler to reason about, and openly documented; Mixpanel-style attempt semantics are deferred until totals counting exists.
- D9 — Sessions never split at project-midnight (Mixpanel resets all sessions at midnight project time). WHY: a visit spanning midnight is one visit; the arbitrary cut exists for Mixpanel's partitioning convenience, not the user's mental model; we keep only the 24h max-duration cap and document the near-midnight delta.
- D10 — Funnel default entry = best-anchor (the user's furthest run counts; ties → earliest — Amplitude 'earliest and longest'-compatible; internal/funnel/funnel.go:152-179) instead of Mixpanel's default first-Step-1-locks-the-window. WHY: first-touch-lock silently zeroes users who retried and later converted — a notorious footgun Mixpanel itself added Optimized Re-entry to escape; changing our shipped default would also violate STABILITY.md. entry=first ships for exact Mixpanel-default parity in parallel runs.
- D11 — Dedup is exact-tuple (name, timestamp, distinct_id, insert_id), unbounded lookback, deterministic last-write-wins (Mixpanel layers a same-calendar-day compaction pass that is 'not guaranteed' and 'in some cases may not run at all'). WHY: nondeterministic dedup is untestable and unauditable; ours is a fixed contract a CI test can pin forever.
- D12 — Unknown event names and enum values are 400 errors with hints, never empty 200s (Mixpanel returns zeroes for nonexistent events; PostHog silently rewrites WAU→DAU at coarse grains). WHY: a typo must look like a typo, not like churn; silent semantic rewrites are how tools lose trust.
- D13 — Empty measure buckets are null, not 0 (PostHog coerces ifNull(...,0) and its own code comment concedes it is 'not quite kosher' for min/max). WHY: 0 is a measurement; absence is absence; conflating them corrupts averages and minimums downstream.
- D14 — No plan-gated correctness or egress: raw export, warehouse-bound pipelines-equivalents, alerts, and cohort-sync webhooks all live in the OSS core with no cadence gating (Mixpanel gates Data Pipelines/alerts behind Growth+/add-ons; Amplitude gates real-time cohort sync to Growth/Enterprise). WHY: those gates are the documented churn triggers of the incumbents — replicating them would rebuild the thing people are fleeing.
- D15 — No query-time governance that silently alters results: no Lexicon-style HIDE that excludes events from query RESULTS or MERGE that rewrites event names at query time (both Mixpanel behaviors). We offer explicit block-at-ingest and audited deletion only, and query results always reflect stored events. WHY: for a 1-20-person team, a governance layer that changes numbers without touching data is a debugging nightmare that contradicts a receipts-based product.
