# smolanalytics analysis-depth build spec (2026-07-14)

## gap verdict

The founder feeling is accurate at the INTERACTION layer, not the engine layer. Ranked by how much each explains "free tiers beat us": (1) No previous-data access — his literal words. Mixpanel/Amplitude/PostHog free all have full date pickers (since/last-N/custom/hourly) plus a compare-to-past ghost overlay with % change; we expose 7/30/90 chips and compare only in ask-verdict prose. (2) One-filter-at-a-time digging — all incumbents ship an unlimited multi-row AND/OR filter builder with operator dropdowns and seen-value autocomplete on EVERY view; our engine has 9 ops but the UI exposes one explore filter row and eq-only chips. (3) No datapoint-to-people drill — Amplitude Microscope / Mixpanel View Users / PostHog persons-modal turn every aggregate into a click-to-the-actual-users; we have zero, despite /v1/users/{id} and cohorts making it pure wiring. (4) One event per chart — no multi-event overlay, no uniques/WAU/MAU/frequency measure switching, no A/B formulas, so ratio metrics (activation rate, ARPU) are inexpressible. (5) Funnel/retention shallowness — no exclusion steps, per-step filters, order modes, hold-constant, time-to-convert view; retention birth event is hardwired to first-ever event. (6) Cohort criteria — no did-NOT-do, counts, time windows, sequences. (7) No browsable people directory or filterable raw-event stream. Honest flip side: we already beat their free tiers on saved cohorts (Mixpanel can't save them free, Amplitude paywalls them entirely), webhook threshold alerts (Mixpanel paid-only), unlimited saved insights (vs 5/10 caps), p90 property measures, default dev-traffic exclusion, and the ask/verdict layer nobody has. The engine holds most primitives; the dashboard surfaces maybe 30% of them. Closing gaps 1-3 alone flips the comparison.

## build spec (ranked)

### 1. Universal FilterSet — one filter system on every view, in the URL
**spec:** Extend internal/query with FilterSet: {groups:[{op:'and'|'or', rows:[{property, op, values[]}]}]}; groups AND together, rows join by group op. Add ops: regex, nregex, ncontains, before/after (date) to the existing 9; multi-value on eq/contains (OR within property, generalizing 'in'). Back-compat: flat filters[] parses as one AND group; dev-traffic exclusion default stays in Apply. Every /v1 endpoint switches filtersFrom(r) to filterSetFrom(r). URL grammar (the shareability constraint): repeatable ?f=<prop>:<op>:<v1|v2|...> rows + ?fm=all|any match mode (Umami-style); two-level groups via JSON body for API/MCP only. /v1/meta extended to serve per-event property names + top-20 seen values (last 30d) for typeahead. UI: ONE shared vanilla-JS filter-row component in the single dashboard file, rendered on every tab: property typeahead, operator dropdown, multi-value input with seen-value autocomplete, '+ filter' adds rows, all/any toggle. Chips render the operator as a click-to-cycle segment (eq -> neq -> contains -> regex, stolen from Fathom). Collapsed to the existing quiet chip bar by default; the builder opens on '+ filter' (progressive disclosure). Ask-bar compiles questions into the same FilterSet and shows it in the computed_by receipt.
**engine:** internal/query: FilterSet type + group matcher + new ops (Go regexp, date compare) + Validate extension; internal/api: filterSetFrom(r) parsing both grammars; /v1/meta gains property/value catalog; all /v1 handlers swap to FilterSet
**ui:** shared filter-row component (one JS function + template partial), operator-cycling chips, typeahead against /v1/meta, ?f/?fm serialization everywhere
**beats them:** URL-as-complete-state (only Umami does this) so every deep-dive is pasteable into Slack; the op-cycle chip makes negation one click with no modal; and the same grammar is the ask-bar's and MCP's compile target, so human, agent, and receipt all speak one provable query language — no incumbent unifies these

### 2. Compare-to-past + annotations (the context layer)
**spec:** Add compare=prev|prev_year|custom:<start> to /v1/trends and /v1/web: response gains previous:{series, total} aligned onto the current x-axis plus delta_pct per point and overall; divide-by-zero yields null. computed_by receipt names BOTH windows explicitly. Annotations: new small store (id, date, text, scope global|insight, author, created_at) with CRUD at /v1/annotations and range-filtered GET. UI: dashed ghost series on the chart, delta badge on every glance tile ('+23% vs prior 30d'), absolute/% toggle; annotation markers sit quietly on the x-axis, hover shows the note, clicking a chart date opens a one-field add box. Verdict line always cites the prior window it compared against.
**engine:** compare param + aligned prior-window computation in internal/trends and internal/web; new internal/annotate store + /v1/annotations CRUD
**ui:** ghost dashed series + delta badges + abs/% toggle in the existing chart renderer; x-axis markers with hover + inline add
**beats them:** incumbents show you the ghost line; our verdict layer TELLS you whether the delta matters, in prose, with computed_by receipts for both windows — and a webhook/deploy hook can auto-annotate releases, which none of them do free

### 3. Real date control — since/last-N/custom/granularity, in the querystring
**spec:** Centralize window parsing (generalize parseTrendWindow in query_api.go into a shared timewindow helper used by every /v1 endpoint): range=today|yesterday|7d|14d|30d|90d|180d|ytd|all|since:<date>|<start>_<end>, plus granularity=hour|day|week|month for bucketed endpoints (hourly capped at 31 days, Mixpanel's guardrail). UI: keep 7/30/90 chips as the calm defaults; a 'more' chip opens a small popover with presets, a since-date field, a two-date custom picker, and the granularity switch. Selection writes range= and gran= into the querystring next to ?f so dated views share like everything else.
**engine:** shared timewindow parser + granularity param threaded through trends/web/funnel/retention bucketing (mostly plumbing over existing start/end support)
**ui:** popover date picker (native <input type=date>, no library), granularity switch, chip state from URL
**beats them:** the range grammar is human-typeable and agent-usable (ask-bar compiles 'since july 1' to range=since:2026-07-01); incumbents' pickers are UI-only state that dies when you share the link

### 4. The people behind every number (our Microscope)
**spec:** Every aggregate response gains who: a persons-descriptor — the exact {event, FilterSet, range, position} that reproduces that cell (trends point=date, funnel step=N&state=converted|dropped, retention cell=bucket:interval, breakdown row=prop:value). New GET /v1/who accepts a descriptor and returns [{user_id, key traits, event_count, last_seen}] with paging + CSV. POST /v1/cohorts accepts {from_who: descriptor} to snapshot a static cohort. UI: clicking any chart point, funnel bar, retention cell, or breakdown row opens ONE shared slide-over panel: the user list, each row expanding inline to the /v1/users/{id} timeline, plus 'save as cohort', 'filter to these', and CSV buttons. Panel header renders the descriptor as its computed_by receipt. Panel state serializes to ?who=<descriptor> so even a drill-through is a shareable URL.
**engine:** who-descriptor emitted by trends/funnel/retention/breakdown responses; new /v1/who resolver (re-runs the slice, collects distinct users); from_who static-cohort path in internal/cohort
**ui:** one slide-over persons panel component, click handlers on chart/table/funnel/retention surfaces, ?who= URL state
**beats them:** the verdict layer auto-reads the slice ('12 dropped users, 9 are plan=free, 7 hit error_shown next') instead of just listing bodies; CSV export of the list is free where Amplitude gates it Growth+; and the drill state itself is a URL, which nobody has

### 5. Multi-series trends with measure library and formulas
**spec:** /v1/trends accepts series=[{event, filters(FilterSet), measure}] (max 6, lettered A-F) alongside the legacy single-event params. Measure library grows to: total, unique_users, unique_sessions (session_id, else 30-min gap), wau, mau (rolling), per_user:avg|median|p90, distinct:<prop>, and min/max/p95/p99 joining existing sum/avg/median/p90 over a property. formula=<expr> evaluates + - * / with parens over the letters per time bucket, server-side in Go (shunting-yard, deterministic); divide-by-zero returns null points, never 0. Receipt includes each series' query plus the formula string. UI: legend under the chart with click-to-hide per series; a quiet '+ compare event' button adds series (progressive disclosure — one event stays the default view); per-series measure dropdown; formula input only appears once 2+ series exist. URL: s=A:signup:unique_users&s=B:purchase:unique_users&formula=B/A.
**engine:** series composition + new measures in internal/trends (all cheap over stored events); formula evaluator (~100 lines); receipt extension
**ui:** multi-line chart with legend toggles, series/measure/formula controls behind '+ compare event', s=/formula= URL grammar
**beats them:** formulas free (Amplitude paywalls them at Plus+), and the ask-bar compiles 'signup to paid rate weekly' straight into a two-series formula query — the exact thing incumbents make you hand-assemble letter by letter

### 6. Funnel + retention truth pack
**spec:** Funnel (internal/funnel): steps become [{event, filters(FilterSet)}] (per-step inline filters); exclusions=[{event, between:[i,j]}]; order=strict|sequential|any; hold_constant=<prop> (all steps must share the value); two derived views in the response: time_to_convert histogram per step-pair and conversion-over-time (rate by entry date); breakdown segments gain a two-proportion z-test significance flag (marked 'insufficient' under 30 entries). Counting stays uniques, documented in the receipt. Retention (internal/retention): start_event + return_event params (birth = first start_event, replacing the hardwired first-ever-event), mode=on|on_or_after (aliasing rolling), custom brackets=0,1-3,4-6,7-13, breakdown param, and who-descriptors on every grid cell. UI: funnel builder rows get a '...' per-step filter (reusing the shared component), an exclusion row, and an order toggle; small tabs on the funnel card: steps / over time / time to convert. Retention gets start/return pickers and a brackets field. All params in the querystring.
**engine:** order modes, exclusions, per-step FilterSet, hold_constant, two derived views, z-test in internal/funnel; start/return split, brackets, breakdown, cell descriptors in internal/retention
**ui:** per-step filter affordance, exclusion + order controls, funnel view tabs, retention start/return/bracket inputs
**beats them:** significance chips plus our existing honest incomplete-bucket labels mean every conversion number carries its own trust markers with computed_by receipts — incumbents let a 12-user segment or wrong-window funnel silently mislead

### 7. Behavioral cohorts v2 — the paywalled thing, free and unlimited
**spec:** Extend internal/cohort's model to criteria groups (and/or) of clauses: {did|did_not, event, count_op gte|lte|eq, count, window: last_n_days|between:<dates>|within_first_n_days-of-first-use}, property clauses via FilterSet, and {sequence: did X then did|did_not Y within N days}. Evaluated deterministically at query time over the event slice, so 'dynamic recompute' is free by construction. Cohorts become usable in three positions: as a FilterSet row (cohort:<id>), as a breakdown dimension (breakdown=cohort:<id1>,<id2> renders in/out or multi-cohort series), and as static snapshots via from_who (item 4). Unlimited saved. UI: a cohort builder view linked from the filter row's cohort option — clause rows reuse the shared filter component plus did/did-not selector, count, and window fields; live size preview as you edit; each saved cohort displays its definition JSON as its receipt.
**engine:** criteria model + evaluator in internal/cohort (did-not, counts, windows, sequence); cohort refs inside FilterSet resolution; cohort-as-breakdown in internal/query
**ui:** cohort builder view reusing filter rows, live size preview, cohort option in filter row and breakdown menus
**beats them:** Mixpanel free users can build but not SAVE cohorts; Amplitude paywalls saved cohorts entirely at Plus+ with a 5-20 cap. We ship unlimited saved behavioral cohorts with negation, counts, and sequences, free, each carrying its definition as a provable receipt — a direct wedge at both incumbents' most-resented gate

### 8. People directory + live events explorer (the trust surfaces)
**spec:** GET /v1/users (list): q= search over user_id and trait values, FilterSet over traits, cohort= scope, sort=last_seen|event_count, paging. GET /v1/events/recent upgraded: FilterSet, event= name filter, cols= property picker, since_id cursor for polling. UI: People tab — table with an 'edit columns' picker, rows click through to the existing /v1/users/{id} timeline, 'save this filtered list as a cohort' button. Events tab — live table polling since_id every 5s with a pause control, rows expand to pretty-printed raw JSON payload, user-id cells link to the profile. Both tables' full state (filters, columns, sort) lives in the querystring.
**engine:** /v1/users list endpoint over the store's user index; FilterSet + cols + since_id on /v1/events/recent
**ui:** two tables sharing the filter component, column picker, JSON row expand, 5s poll with pause
**beats them:** this is the verify-surface for our MCP instrumentation USP: the agent that just wrote your tracking watches its own events land, filtered to exactly what it shipped — documented as a step in the instrumentation flow. Amplitude gates user CSV export behind Growth; ours is free
