# Feature build plan — the 5 deferred features

Generated from a codebase-grounded design pass (2026-07). Ordered by build sequence.

## Feature flags — the full API/MCP/SDK/dashboard surface on top of the existing internal/flag engine core
**Effort:** ~7 eng-days

The engine core (types + deterministic FNV eval + JSON upsert store) already exists at internal/flag; what ships is the surface around it, mirroring exactly how cohorts and deploy markers are wired. Server-side evaluation is the source of truth: agents/dashboard do CRUD over MCP (read-key-authed writes) and /v1, and a public write-key GET /v1/flags/evaluate returns each user's resolved key→variant map so the browser+mobile SDKs fetch once and read smol.flag(key, default) synchronously from cache. "Measured" flags add an A/B results report (per-variant conversion + delta) computed by a pure flag.Measure() function pinned byte-for-byte across /v1 and MCP by an agreement test, fed by $feature_flag_called exposure events the SDK emits once per user/session over the existing /v1/events path.

**Fits philosophy:** Stays fully inside the single-binary model: no new storage engine — flags persist in .flags.json via the existing flag.Store (sync.Mutex + atomic tmp+rename + List() returns a copy), identical to cohort/store.go and deploys. No new ingestion path — exposures ride the existing batched POST /v1/events at ~7 bytes/event, and only for flags explicitly marked Measured, so ordinary flags add zero events (the exact intent noted in flag.go's Measured comment). Evaluation is deterministic Go FNV-1a done server-side; SDKs cache the RESOLVED values rather than re-implementing the hash, so there is no cross-language bucketing-drift risk and no need to ship targeting rules to browsers. The measured report reuses the same event scan the dashboard uses and is pinned MCP==/v1 by a new agreement test, honoring the CI contract. Strain point: GET /v1/flags/evaluate must be reachable by the public write key (browsers only hold that key), which is a deliberate carve-out in the auth model — mitigated by returning ONLY resolved key→variant for the requested distinct_id (never rule definitions or other data), exactly the exposure PostHog's /decide already accepts for client flags.

**Data model:** Reuses internal/flag types unchanged: Flag{Key, Description, Enabled, Variants []Variant{Key,Weight}, Rules []Rule{Filters []query.Filter, RolloutPct}, Measured bool, Created, Updated} keyed by Key (Save is an upsert, unlike cohort's random-id create). Persisted at <dataPath>.flags.json. Evaluate(distinctID, context) already returns (variant, on). New pure type in internal/flag/measure.go: Report{Flag string, Goal string, Days int, Variants []VariantResult{Key, Exposed, Converted, RatePct, DeltaPct, Significant, SmallSample}, Control string, Note string} where DeltaPct/Significant compare each variant to the control arm via a two-proportion z-test, small-sample-guarded (n<~30) like the r5b regression tests. Exposure record: event name "$feature_flag_called", properties {"$feature_flag": key, "$feature_flag_response": variant} — the "$" prefix means control.go already exempts it from track-plan "unplanned event" drift (internal/mcp/control.go:313), so no whitelist edit is needed.

**API:** GET /v1/flags — list all flags (read-key or session, via authMW like GET /v1/cohorts), POST /v1/flags — upsert a flag from a Flag JSON body (session-only write, parity with saveCohort), DELETE /v1/flags/{key} — delete by key (session-only write), GET /v1/flags/evaluate?distinct_id=&context= — PUBLIC, write-key-authed via s.ingestAuth + CORS; returns {flags:{key:variant}, measured:[keys]} for that user, OPTIONS /v1/flags/evaluate — CORS preflight (browser GET with Authorization header), GET /v1/flags/{key}/measure?event=&days= — per-variant A/B results (read-key or session; agreement-pinned to MCP flag_impact)
**MCP tools:** create_flag — key, description, enabled, rollout_pct, variants[{key,weight}], rules[{filters,rollout_pct}], measured; upserts via flag.Store.Save, list_flags — flags with enabled/rollout/variant summary, set_flag_enabled — key, enabled; the flip, via flag.Store.SetEnabled (deploy-marker hook point in increment 6), delete_flag — key (dedicated handler, NOT deleteByID which expects id), evaluate_flag — key (optional=all), distinct_id, context; resolved value(s), byte-for-byte equal to GET /v1/flags/evaluate, flag_impact — key, event(goal), days; the measured report, pinned to GET /v1/flags/{key}/measure (beyond the required 5, needed for the measured feature + agreement test)
**SDK:** Browser internal/api/sdk.js: on init (opts.flags !== false) fetch host+"/v1/flags/evaluate?distinct_id="+distinctId() with Bearer write key, cache {flags, measured}; add smol.flag(key, default)=synchronous cache lookup (returns default until loaded), smol.flags(), smol.onFlags(cb); re-fetch on identify(); first read of a measured key enqueues $feature_flag_called {$feature_flag,$feature_flag_response} once/session (dedupe Set). The 4 native SDKs (android SmolCore.kt, swift SmolCore.swift, react-native src/core.ts, flutter lib/src/smol_core.dart) get the same evaluate-fetch cache + flag(key,default) + measured-exposure emit in the shared core, caching resolved values (no client-side bucketing). Public API additions surfaced in each SDK's facade (SmolAnalytics.kt/.swift, index.ts, smolanalytics.dart).
**UI:** Cloud project page (app/(app)/projects/[id]): new Flags.tsx section modeled on CohortBuilder.tsx — a list of flags each with an enabled toggle, rollout% slider, and variant/weight chips; a create/edit form for key + description + enabled + rollout% + variants(+weights) + targeting rules (property/op/value, reusing the /meta event+property pickers) + a "measured" checkbox; and for measured flags a results panel modeled on DeployImpact.tsx showing per-variant exposed/converted/rate and the delta vs control with a significance badge. All writes go through app/api/projects/[id]/flags/route.ts, which proxies to the instance /mcp with the read key (create_flag/set_flag_enabled/delete_flag), mirroring cohorts/route.ts; the measured read proxies to flag_impact. Engine dashboard.tmpl.html gets an optional read-only flag list for self-hosters (deferred).
**Storage:** .flags.json via existing internal/flag.Store (atomic tmp+rename, mutex, List-copy); exposures in the normal columnar segment store; measured report computed on-demand from one event scan. No new DB/queue.

**Risks:**
- Public write-key on GET /v1/flags/evaluate is a deliberate auth carve-out (browsers only hold the write key). It must return ONLY the resolved key→variant map for the requested distinct_id — never Rules/Filters/definitions or other flags' internals — or targeting logic leaks client-side. Anyone can enumerate distinct_ids to map bucketing, which is inherent to client-side flags (same as PostHog /decide); document it, don't try to hide it.
- CORS: evaluate is a cross-origin browser GET carrying an Authorization header, so it triggers a preflight. The existing setCORS() advertises only 'POST, OPTIONS' — it must be extended (or a GET-specific header set) and an OPTIONS /v1/flags/evaluate route added, or the browser call fails silently with no events and no flags.
- Exposure inflation: emitting $feature_flag_called on every evaluate would explode event count and violate the ~7-bytes-per-event ethos. Mitigation is firm: server GET stays side-effect-free; the SDK emits exposure only for Measured flags, once per distinct_id+variant per session (in-memory dedupe). Get this wrong and a busy page doubles its event volume.
- Determinism/agreement drift: server Go FNV-1a is the source of truth and SDKs cache resolved values, so there is no cross-language hash to keep in sync — BUT if any mobile SDK later adds offline client-side bucketing it must reproduce hash_fnv1a(salt+':'+id) byte-for-byte; keep mobile on cached-values in v1 to sidestep this entirely.
- Key vs id: the flag store is keyed by Key with Save-as-upsert (not cohort's random id + create-only). Routes are /v1/flags/{key}, and delete_flag/set_flag_enabled take {key} — do NOT blindly reuse the MCP deleteByID helper, which unmarshals {id}. create_flag on an existing key silently overwrites; the UI/tool should surface that it's an update.
- Measured report correctness: only count a goal conversion that occurs AFTER the user's first exposure timestamp (else you measure pre-existing behavior), and small-sample-guard so a 1/1=100% arm is not headlined as a winner (reuse the r5b small-sample pattern). Two-proportion significance on tiny n is meaningless — flag it, don't assert it.
- create_flag must accept nested targeting (query.Filter rules + variants+weights) via the same FilterSet parsing the other MCP tools use, or agents can only produce boolean/rollout flags and the targeting half of the product is unreachable from the editor.
- Four mobile SDKs multiply surface area and several lack CI here (swift/flutter build locally). Budget real build/verify time per SDK; a thin flag(key,default) is easy, but 4x compile-and-smoke is where the day goes.

## ab-experiments
**Effort:** ~8.5 eng-days

An experiment binds an existing feature flag (deterministic FNV variant assignment, already in internal/flag) to a goal event, tracks a reserved `$exposure` event as the denominator, and computes a per-variant conversion report with uplift + a two-proportion z-test / confidence in pure Go (stdlib math.Erfc). It ships as a new internal/experiment package exposed identically over GET /v1/experiments/results and an MCP `experiment_results` tool (pinned byte-for-byte by the agreement test), plus create/list/delete on both surfaces, a `/v1/decide` assignment+exposure endpoint, and an "Experiments" tab in the engine dashboard and cloud project page.

**Fits philosophy:** Stays true to the single-binary identity: the only new math is Go stdlib `math.Erfc` (verified: Φ(z)=0.5·erfc(-z/√2) gives z=2.10/p=0.035 for a 10→13% lift) — no stats library, no service. Two config stores (.experiments.json, .flags.json) are JSON sidecars using the exact atomic tmp+rename + sync.Mutex + List()-returns-a-copy pattern from internal/flag/store.go and internal/cohort/store.go. Results are computed on-read from the same columnar event stream as every other report (no materialized aggregates, nothing new at ~7 bytes/event). CI byte-equality holds by construction because both /v1 and /mcp call ONE shared experiment.Resolve, and floats are rounded to fixed precision before marshal. Strain 1: exposure events add event volume (and billing is per-event) — mitigated by the flag's already-reserved `Measured` opt-in (only measured flags emit `$exposure`) plus once-per-session client-side dedup. Strain 2: float/JSON determinism across surfaces — mitigated by the single-function design + fixed rounding, and enforced by extending agreement_test.go. Strain 3: it drags feature-flags (currently core-only, wired nowhere) into the wired surface — acceptable and intended, since flags ARE the assignment mechanism; kept minimal (create_flag/list_flags only).

**Data model:** New internal/experiment package. Experiment{Key string (stable id + bound flag key); Name; Flag string (flag that assigns variants, defaults to Key); Goal, GoalKind ("event"|"path"), GoalValue (conversion, mirrors goal.Definition so semantics never drift — see reuse note); Control string (baseline variant, sets uplift sign); Stopped bool; Created, Updated time.Time}. Reuses flag.Flag verbatim for variants + deterministic bucketing (flag.Variant{Key,Weight}). Reserved event: `$exposure` with properties{flag, variant} — the denominator. Report structs: VariantResult{Variant, Exposed int (unique users bucketed), Converted int (exposed users who hit the goal AT OR AFTER their first exposure), Rate float64, UpliftPct float64, Z float64, PValue float64, Confidence float64, Significant bool, IsControl bool} and Report{Experiment, Flag, Goal string, PeriodDays int, Control string, Variants []VariantResult, TotalExposed int, MixedAssignment int (users seen on >1 variant, excluded), SRMWarning bool, Winner string, Note string}. Reuse: refactor goal.go's unexported matches into an exported goal.Match(e,kind,value) called by both goal.Resolve and experiment.Resolve so "what counts as a conversion" is one definition.

**API:** GET /v1/experiments/results?experiment=<key> | ?flag=&goal=&goal_kind=&control=&days= (read-key; agreement-pinned), GET /v1/experiments (list; read-key), POST /v1/experiments (create/bind flag+goal+control; session), DELETE /v1/experiments/{key} (session), GET /v1/flags (list; read-key), POST /v1/flags (create/upsert flag with variants; session), DELETE /v1/flags/{key} (session), POST /v1/decide (assign variants for enabled+measured flags and log $exposure server-side; write-key) [Inc3]
**MCP tools:** experiment_results, create_experiment, list_experiments, delete_experiment, create_flag, list_flags
**SDK:** Inc3 adds three methods to internal/api/sdk.js (the browser autocapture script): smolanalytics.decide(context) POSTs {distinct_id, context} to /v1/decide, caches the returned {flags:{key:variant}} map, and relies on the server to log $exposure (guarded once per session via the existing session object so pageview churn doesn't re-emit exposures); smolanalytics.variant(key) returns the cached assignment for app-code branching; smolanalytics.exposure(flag, variant) is a raw escape hatch that enqueues a {name:"$exposure", properties:{flag,variant}} event for teams that assign variants themselves. No bucketing logic is copied into the SDK — assignment stays server-authoritative to avoid desync. Native SDKs in /Users/arjun/smolanalytics-cloud/sdks/ can add the same thin exposure() helper later; they already POST to /v1/events.
**UI:** Engine dashboard (dashboard.tmpl.html): an "Experiments" section modeled on the existing Goals/Cohorts blocks — one card per experiment showing per-variant conversion bars, exposed/converted counts, uplift vs control, a confidence value and a significance/"not enough data yet"/SRM-warning badge, with a create form (name, flag+variants, goal, control). Cloud project page: a new Experiments.tsx rendered as a tab inside Reports.tsx (alongside traffic/funnel/retention), fed by app/api/projects/[id]/experiments/route.ts which lists + resolves results and creates experiments through the instance /mcp using the read key (never exposing it to the browser), mirroring CohortBuilder + cohorts/route.ts.
**Storage:** Two JSON sidecars alongside the event log, opened in cmd/smolanalytics/main.go for both `serve` and `mcp` exactly like .goals.json/.cohorts.json: .experiments.json (experiment.Store) and .flags.json (flag.Store, currently unopened). No aggregates are stored; the results report is recomputed from the columnar event segments on every read, same as funnel/goal/retention.

## Click heatmaps — page-level click-density grid + top clicked elements, agent-native (MCP) and dashboard overlay
**Effort:** ~6 eng-days

Extend the existing `$click` autocapture to carry compact viewport width + scroll offset (reusing the clientX/clientY it already sends), then compute a deterministic click-density grid per page at query time — no new stored config, no screenshot service. Ships as a pure `internal/heatmap` engine behind `GET /v1/heatmap` and a byte-identical MCP `heatmap` tool, then a dashboard `HeatmapView` that overlays a density canvas on an iframe of the live page (auto-falling back to a grid-only "blueprint" when the site blocks framing).

**Fits philosophy:** Stays fully inside the single-binary identity: (1) NO new store — heatmaps are a query-time aggregation over existing `$click` events exactly like `internal/paths` and `internal/web`, so the cohort/flag Store pattern is deliberately NOT used (that pattern is for saved *config*, and a report needs none). (2) Screenshot service is REJECTED — a static Go binary can't ship headless Chromium; the page image comes from an iframe of the live URL rendered browser-side in the cloud app, and the engine only ever returns integer coordinates, never pixels/images. (3) ~7 bytes/event honored: no new normalized field is stored — the SDK reuses the `x`/`y` (clientX/clientY) it ALREADY sends on `$click` and adds only `vw` (innerWidth, int) always plus `sy` (scrollY, int) when >0; short keys + small ints that flate collapses across a segment (codec.go dictionary+flate), so per-click raw growth ~10-15B compresses to ~1-2B. Strain point: the headline "~7 bytes" is a post-compression average, and click-heavy pages nudge it up — mitigated by the tiny-int scheme and by the fact that only `$click` events grow.

**Data model:** See data_model field (no new store; $click gains vw+sy; heatmap.Result = Cells grid + TopTargets).

**API:** GET /v1/heatmap?path=/pricing&viewport=desktop&cols=40&row_px=20&days=30&filters=... (read-key auth, mirrors GET /v1/paths: parse params, s.filtered(r), window-scope, heatmap.Compute)
**MCP tools:** heatmap (read tool; description: 'Where users click on a given page — a click-density grid plus the top clicked elements, for building a heatmap. Requires a path (get one from web_overview top_pages); optional viewport=mobile|tablet|desktop, cols, row_px, and the standard days/hours/from/to/filters window.'). Registered in internal/mcp/tools.go toolList, dispatched in internal/mcp/mcp.go callTool switch returning jsonText(heatmap.Compute(...)) — byte-identical to GET /v1/heatmap.
**SDK:** internal/api/sdk.js only: in the existing click handler's `props` object add `vw: window.innerWidth` and, guarded, `if (window.scrollY) props.sy = Math.round(window.scrollY)`. Reuses the already-captured `x`(clientX), `y`(clientY), `path`. No init/API surface change, no wire-version bump. Native/mobile SDKs untouched (no click coords). ## session-replay
**Effort:** ~14 eng-days

A default-OFF, opt-in-per-instance session replay: a lazily-loaded browser recorder (vendored rrweb-record, NOT bundled into the ~7KB core SDK) uploads gzipped rrweb-event chunks to a new POST /v1/replay endpoint; chunks land as gzipped files in the EXISTING internal/store/blob.Blob (Local files / S3, same interface the columnar segments already use — no new storage engine), while lightweight per-session metadata lives in a new JSON index store that copies the cohort/flag store pattern. A dashboard Replays tab lists sessions (duration, pages, rage-clicks) and a dynamically-imported rrweb-player streams the chunks. Sampling is client-side + a hard server-side disk-MB budget with LRU eviction, so replay never starves the event log.

**Fits philosophy:** Honest verdict: replay is the ONE feature that breaks the ~7-bytes/event promise at the payload level — a recorded session is ~150KB gzipped (~20,000x heavier per unit than an event), and no amount of cleverness makes DOM mutations 7 bytes. But it fits the parts of the identity that actually define the product: (1) SINGLE STATIC BINARY, NO EXTERNAL DB/QUEUE — chunks are just gzipped files written through the blob.Blob interface that ALREADY exists (internal/store/blob/blob.go: Local with atomic tmp->fsync->rename, plus stdlib-only S3/R2), the same seam the segment cold-tier uses; zero new infra, no ClickHouse, no Kafka. (3) OPT-IN PER INSTANCE, DEFAULT OFF via SMOLANALYTICS_REPLAY (unset => endpoint 404s, SDK module never loads, zero disk, zero SDK weight) — so the 95% who never enable it keep the lean identity fully intact; you only pay disk when you flip the switch, and the cloud gates it to paid plans. **Risks:**
- Disk exhaustion on the shared 1GB volume starving the event log (the actual product) — mitigate with the hard MAX_MB budget + LRU eviction + default-OFF + gate to paid plans; replay never gets to compete with segments for space.
- PII/breach liability: replay is the single highest-risk surface in the whole product — one unmasked password/email field is a data breach. Mitigate: mask-all-inputs AND mask-all-text ON by default, masking at CAPTURE (server never receives raw text), hardcoded block of password/email/file inputs, DNT/consent respect, and GDPR erase wired through delete_user_data. This must be right in Inc 1, not deferred.
- rrweb fidelity/version drift: complex DOMs (shadow DOM, cross-origin iframes, canvas/WebGL, web components, CSS-in-JS) record imperfectly or bloat; vendor a PINNED rrweb version (no CDN — matches self-host ethos), document limitations, and set a per-chunk size cap so a pathological page can't send 50MB.
- Bundle weight regression: naively bundling rrweb-record into sdk.js would ~6x the ~7KB core SDK — the design MUST keep it a separate lazily-injected /sdk-replay.js, and rrweb-player MUST be a dashboard-only dynamic import; a careless import breaks the lean-SDK promise.
- Agreement-test friction: the MCP==/v1 byte-for-byte rule applies to reports; the binary chunk stream cannot be JSON-DeepEqual'd — structure it so ONLY GET /v1/replays vs list_replays is agreement-tested and the stream is exempt exactly like /v1/export, or CI fails.
- Ingest amplification + cost: replay POSTs are far larger/more frequent than event batches (per-session chunk cadence, keepalive floods on unload) and, on the S3 backend, add a PUT per chunk + GET-egress per playback; bound with the 5s/100-event/500KB flush cadence now and coalesce chunks later.
- Server-side derivation trust: rage-click and page counts must be computed at ingest from the rrweb stream, not read from client-supplied fields, or the LIST report (and any alert built on it) is spoofable via the public write key.
